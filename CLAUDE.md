# CLAUDE.md — LevelUp Academy

## Что это за проект

**LevelUp Academy** — SaaS-платформа (CRM) для управления учебными центрами в Узбекистане. Продаётся партнёрам (учебным центрам) как готовое решение: каждый партнёр — отдельный тенант со своей организацией, филиалами, сотрудниками и студентами.

### Зачем создан

- Учебные центры в Узбекистане работают вручную или на примитивных таблицах
- Нужна единая система: учёт студентов, посещаемость, тесты, ДЗ, финансы (доход/расход/платежи), коины (геймификация), чаты, Telegram-уведомления
- Платформа масштабируется: один бэкенд обслуживаетundreds партнёров (мульти-аренда)

### Что умеет

- Приём заявок с лендинга → онбординг партнёра → создание организации
- Управление филиалами, сотрудниками, студентами, группами
- Финансы: платежи, сплит-платежи, расходы, отчёты
- Геймификация: коины за достижения, магазин, лидерборды
- Тесты, ДЗ, видео-уроки
- Realtime-чаты (Socket.io), Presence (онлайн)
- Telegram-бот для уведомлений

## ⛔ СТРОГОЕ ПРАВИЛО: Frontend ≠ Backend

Это **АБСОЛЮТНОЕ** правило — нарушение = ломает весь проект:

| Зона | Кто трогает | Что НЕЛЬЗЯ трогать |
|------|-------------|-------------------|
| **Backend** (`backend/`) | ТОЛЬКО Karis + Abdulaziz | ❌ Никаких изменений в `frontend/` |
| **Frontend** (`frontend/`) | ТОЛЬКО команда фронта | ❌ Никаких изменений в `backend/` |

### Почему это критически важно

- Бэкенд и фронтенд разрабатываются **разными людьми** (Karis + Abdulaziz на бэке, команда из 10+ человек на фронте)
- Они в **разных ветках** и могут работать одновременно
- **Смешивание кода = конфликты + ломает рабочий процесс + теряет деньги**
- Backend-код влияет на безопасность, данные, платёжную систему — левый человек не должен его трогать

### Как применять

1. **Зона определяется ответом на вопрос «Кто ты?»** (см. раздел «Вход в проект» выше) — отдельно спрашивать frontend/backend не нужно
2. **Backend-люди** — работают ТОЛЬКО в `backend/`, **КАТЕГОРИЧЕСКИ** не заходят в `frontend/`
3. **Frontend-люди** — работают ТОЛЬКО в `frontend/`, **КАТЕГОРИЧЕСКИ** не заходят в `backend/`
4. **Если нужен общий контекст** — читай `docs/` и `.md/` (это общая документация)
5. **Нарушение правила = остановка работы и объяснение руководителю**

### Исключения из правила (кросс-доступ)

Эти две роли — единственные, кто может выходить за рамки своей зоны (`frontend/` ⇄ `backend/`) без нарушения правила выше:

| Роль | Кто | Идентификатор входа | Права |

| **SEO** | Abdulaziz | представляется как `abdulazizSEO` | На время SEO-задач может править любые файлы обеих зон: meta-теги, `sitemap.xml`, `robots.txt`, заголовки/alt-тексты, семантическая разметка, скорость загрузки, SSR/prerender-настройки |

## Архитектурные правила

### Multi-tenancy (мульти-аренда)

- Все таблицы данных содержат `branch_id` и/или `organization_id`
- Каждый запрос фильтруется по `req.scope.organizationId` + `branchId`
- Никогда не запрашивай данные без фильтра по организации

### Финансы

- Сплит-платёж: один `invoice` может быть оплачен несколькими `transaction` с общим `split_batch_id`
- Изменение коинов ТОЛЬКО через `changeCoins()` — никаких прямых UPDATE
- Все внешние интеграции (Telegram, SMTP) — через BullMQ очередь

### Архивация и удаление

- `deleted_at` = soft-delete (фильтровать `deleted_at IS NULL`)
- `is_archived` = read-only (мутации → 403)

## Release

Версия проекта хранится в файле `VERSION` (semver: `MAJOR.MINOR.PATCH`). Релиз = зафиксированная точка в `main`, помеченная git-тегом `vX.Y.Z` + GitHub Release с списком изменений.

**Когда делать релиз:** после того как `save-zone` протестирован и смёржен в `main` (см. workflow в глобальном `CLAUDE.md` пользователя) — то есть релиз это ФИНАЛЬНЫЙ шаг после `save-zone → main`, не замена ему.

```bash
python scripts/release.py          # patch: 0.1.0 -> 0.1.1 (мелкие фиксы)
python scripts/release.py minor    # 0.1.0 -> 0.2.0 (новая фича, без breaking changes)
python scripts/release.py major    # 0.1.0 -> 1.0.0 (крупный этап, напр. конец MVP1)
```

Скрипт сам: требует чистый `git status` и ветку `main` → пуллит → бампит `VERSION` → собирает `CHANGELOG.md` из коммитов с прошлого тега → коммитит, ставит тег, пушит `main`+тег → создаёт GitHub Release (`gh release create`).

### Откат (rollback)

Каждый релиз — это точка возврата. Если после релиза в `main` попало что-то, что всё сломало:

```bash
python scripts/rollback.py v0.1.1   # вернуть main к состоянию релиза v0.1.1
```

Это НЕ `git reset --hard` и НЕ force-push (это общий репозиторий, у команды свои локальные копии — переписывать историю нельзя). Скрипт делает: файлы возвращаются к состоянию тега → это фиксируется НОВЫМ коммитом `revert: rollback main to vX.Y.Z` → пушится обычным способом. История видна всем: было видно, что сломалось, видно, когда откатили. После отката имеет смысл сразу сделать `python scripts/release.py`, чтобы у отката тоже был свой тег.

### Уведомление о push в Telegram

После `git push` в `main` — уведомить через `telegram-mcp` (`send_message`), кратко: что запушено, что исправлено/добавлено, какие файлы затронуты. Кому именно — зависит от того, что попросил пользователь:

- **Нақт (наличные):** Кассир принимает наличные, чек генерируется автоматически

- **Сплит-платёж:** Один счёт может быть оплачен частями (нақт + карта)
- **Безопасность:** Все транзакции через BullMQ очередь, без прямых HTTP-вызовов к платёжным системам
- **Аудит:** Каждая транзакция записывается в `transactions` с `split_batch_id`
